# FamilyChoresBot

**FamilyChoresBot** — это микросервисный Telegram-бот, разработанный на **Java 21** с использованием **Spring Framework**.  
Проект демонстрирует архитектуру асинхронной обработки событий с использованием брокера сообщений, контейнерной инфраструктуры и HTTPS-webhook интеграции с Telegram Bot API.

Основная цель проекта — разделение ответственности между компонентами системы, повышение отказоустойчивости и обеспечение масштабируемости обработки обновлений Telegram.

---

# Архитектура системы

Система построена по микросервисному принципу и разделена на несколько компонентов.

```
Telegram
   │
   │ HTTPS Webhook
   ▼
Traefik (Reverse Proxy + TLS)
   │
   ▼
Dispatcher Service
   │
   │ publish update
   ▼
RabbitMQ
   │
   │ consume update
   ▼
Node Service
   │
   │ read/write
   ▼
PostgreSQL
```

### Поток обработки

1. Telegram отправляет обновление через **Webhook**.
2. **Traefik** принимает HTTPS-запрос и маршрутизирует его к сервису `dispatcher`.
3. `dispatcher` публикует обновление в очередь **RabbitMQ**.
4. `node` получает сообщение из очереди и выполняет бизнес-логику.
5. `node` взаимодействует с **PostgreSQL** при необходимости.
6. Результат обработки отправляется обратно через очередь ответов и доставляется пользователю.

# Компоненты проекта

## dispatcher

Сервис, отвечающий за входящие HTTP-запросы от Telegram.

Функции:

- приём webhook-обновлений
- сериализация и публикация обновлений в очередь RabbitMQ
- получение ответов из очереди
- отправка сообщений обратно в Telegram

Основная задача — выступать **gateway-сервисом**, принимающим внешний трафик.

---

## node

Worker-сервис, реализующий основную бизнес-логику бота.

Функции:

- получение сообщений из очереди RabbitMQ
- обработка команд пользователей
- взаимодействие с базой данных
- формирование ответа
- публикация ответа в очередь

Сервис можно масштабировать горизонтально.

---

## common

Общий модуль, используемый `dispatcher` и `node`.

Содержит:

- доменные **Entity**
- **DAO**
- перечисления (**Enums**)
- общие структуры данных

Это позволяет избежать дублирования кода и поддерживать единый контракт между сервисами.

# Основные архитектурные решения

### Webhook вместо Long Polling

Использование webhook позволяет:

- снизить задержку обработки сообщений
- избежать постоянных polling-запросов
- лучше интегрироваться с production-инфраструктурой

---

### Асинхронная обработка через RabbitMQ

RabbitMQ позволяет:

- разделить приём сообщений и их обработку
- обеспечить буферизацию нагрузки
- масштабировать worker-сервисы
- повысить устойчивость системы

---

### Разделение сервисов

Разделение на `dispatcher` и `node` обеспечивает:

- независимое масштабирование
- упрощённую поддержку
- лучшую модульность

---

### Контейнеризация

Использование Docker обеспечивает:

- воспроизводимость окружения
- изоляцию зависимостей
- упрощённый деплой

---

# Безопасность

Репозиторий не содержит секретных данных.

Следующие параметры хранятся вне репозитория:

- Telegram Bot Token
- данные подключения к PostgreSQL
- данные подключения к RabbitMQ
- домен и TLS-сертификаты

Секреты передаются через переменные окружения.

---

# Наблюдаемость

Сервисы используют логирование для диагностики:

- обработка входящих webhook
- операции очередей RabbitMQ
- взаимодействие с PostgreSQL
- ошибки бизнес-логики

Логи позволяют отслеживать поток обработки событий и диагностировать проблемы в инфраструктуре.
